# Building FlipFinder: Data Ingestion into S3 Vectors

This guide covers the ingestion path used by Frontier retrieval:
1. Upload product JSONL splits to S3
2. Embed product summaries with SageMaker
3. Store vectors and metadata in S3 Vectors

Use this before deploying or validating the Frontier Agent.

## Prerequisites

- Deploy the embedding endpoint first: [EmbeddingEndpoint.md](EmbeddingEndpoint.md)
- AWS permissions for:
  - `s3:GetObject`, `s3:PutObject`
  - `sagemaker:InvokeEndpoint`
  - S3 Vectors operations (`CreateVectorBucket`, `CreateIndex`, `PutVectors`, `QueryVectors`)
- Python dependencies installed from the repo root
- AWS CLI configured for the target account/region

## Step 1: Upload Source Dataset to S3

The upload script is:
- `src/dataset_ingestion/hf_ingestion.py`

It pulls `ed-donner/items_lite` (or `items_full`) and writes:
- `train.jsonl`
- `val.jsonl`
- `test.jsonl`

to your source bucket.

Update these values in `hf_ingestion.py`:
- `LITE_MODE`
- `bucket`

Then run:

```bash
python src/dataset_ingestion/hf_ingestion.py
```

Expected output includes counts for train/val/test and successful S3 upload.

## Step 2: Configure Vector Ingestion

The ingestion script is:
- `src/dataset_ingestion/s3_vector_ingestion.py`

Update the script constants for your environment:
- `AWS_REGION`
- `SOURCE_BUCKET`
- `LITE_MODE`
- `VECTOR_BUCKET`

Notes:
- `SOURCE_KEY` is derived from `LITE_MODE` (`items_lite/train.jsonl` or `items_full/train.jsonl`)
- `INDEX_NAME` is derived from `LITE_MODE` (`products-lite` or `products`)
- The script creates the vector bucket/index if missing

### Optional Runtime Tuning

The script supports these environment variables:
- `SAGEMAKER_ENDPOINTS` (comma-separated endpoint names)
- `SAGEMAKER_ENDPOINT` (single-endpoint fallback)
- `VECTOR_PUT_BATCH_SIZE` (default `128`)
- `VECTOR_EMBED_BATCH_SIZE` (default `8`)
- `VECTOR_MAX_WORKERS`
- `VECTOR_EMBED_MAX_RETRIES` (default `4`)
- `SAGEMAKER_CONNECT_TIMEOUT_SECONDS` (default `10`)
- `SAGEMAKER_READ_TIMEOUT_SECONDS` (default `180`)

PowerShell example:

```powershell
$env:SAGEMAKER_ENDPOINTS = "flipfinder-embedding-endpoint,flipfinder-embedding-endpoint-2,flipfinder-embedding-endpoint-3"
$env:VECTOR_PUT_BATCH_SIZE = "128"
$env:VECTOR_EMBED_BATCH_SIZE = "8"
$env:VECTOR_MAX_WORKERS = "6"
```

## Step 3: Run Ingestion

From repo root:

```bash
python src/dataset_ingestion/s3_vector_ingestion.py
```

Expected log flow:
- Reads `s3://<SOURCE_BUCKET>/<SOURCE_KEY>`
- Creates vector bucket/index on first run (if missing)
- Prints progressive `Ingested ... vectors`
- Finishes with:
  - `Finished ingesting <N> vectors`
  - `Vector bucket: ...`
  - `Vector index: ...`

## Step 4: Validate Semantic Search

Use:
- `src/terraform/FrontierAgent/ingest/test_search_s3vectors.py`

Set environment values first (PowerShell):

```powershell
$env:FRONTIER_AWS_REGION = "us-east-1"
$env:FRONTIER_VECTOR_BUCKET = "products-vectors-<account-id>"
$env:FRONTIER_INDEX_NAME = "products-lite"
$env:FRONTIER_SAGEMAKER_ENDPOINT = "flipfinder-embedding-endpoint"
```

Run:

```bash
python src/terraform/FrontierAgent/ingest/test_search_s3vectors.py
```

This prints sample nearest-neighbor matches and similarity scores.

## Re-ingestion Rules

- Do not mix vectors generated by different embedding models in one index
- If embedding model/dimension changes, use a new index name or rebuild the index
- Keep `FRONTIER_VECTOR_BUCKET` and `FRONTIER_INDEX_NAME` aligned with the ingested target

## Troubleshooting

### `NotFoundException` on bucket/index

Check `VECTOR_BUCKET`, `INDEX_NAME`, and region values in the ingestion script.

### `ModelNotReadyException` or timeout on embedding calls

Increase:
- `SAGEMAKER_READ_TIMEOUT_SECONDS`
- `VECTOR_EMBED_MAX_RETRIES`

and/or reduce:
- `VECTOR_EMBED_BATCH_SIZE`
- `VECTOR_MAX_WORKERS`

### Index dimension mismatch

If the script reports an existing index with the wrong dimension, create a new index name (or recreate the old one) and re-ingest.
